# 产品介绍

## Karpathy LLMWiki 的设计理念

LLMWiki 提出的核心模式是：长期知识需要一个稳定、可维护、会持续增长的 Wiki 表面。LLM 持续把原始资料编译成互相链接的 Wiki 页面。新资料进入时，系统更新实体页、概念页、主题总结、交叉引用和冲突提示，让知识库随着每次入库持续积累。

这个模式里有三层关键结构：

| 层级        | 作用                                                                                     |
| ----------- | ---------------------------------------------------------------------------------------- |
| Raw Sources | 保存不可变的原始资料，作为事实来源                                                       |
| Wiki        | 保存 LLM 生成和维护的 Markdown 页面、总结、实体、概念、关联和综合分析                    |
| Schema      | 约束 Wiki 的结构、命名、链接、frontmatter、入库流程和回答方式，让 LLM 有纪律地维护知识库 |

Wiki 是持续维护的知识资产。LLM 负责归纳、交叉引用、更新和检查一致性；人负责选择资料、提出问题和审查方向。随着每一次入库、查询和整理，Wiki 会越来越完整，查询也可以基于已经沉淀好的页面、链接、索引和历史记录展开。

## FocoContext 是什么

FocoContext 继承了 LLMWiki 的核心理念，并把它做成面向开发者的自部署知识库服务。它把文件、文本、URL、Source Watch 扫描结果和外部提交内容编译成长期可维护的 Wiki 页面、图谱关系、引用、版本记录和可用于检索的上下文包。

它适合放在你的应用后端、Agent 服务或自动化流程旁边，作为独立的知识中间件。你的业务系统负责用户、对话、权限和最终答案；FocoContext 负责保存知识、编译知识、解释来源、返回上下文和维护版本。

## 产品架构

![FocoContext 产品架构](/images/fococontext-product-architecture.png)

FocoContext 的架构理念是知识编译。资料先进入统一 OpenAPI 和任务系统，再经过解析、OCR、Wiki 编译和 Change Set 应用，最终沉淀为 canonical Wiki 页面、知识图谱、来源证据和版本记录。

Fork 是这套架构里的持续更新和隔离增量机制。上层应用或 Agent 可以把用户、工作区或客户确认后的新增内容写入 Fork overlay；系统在 Fork scope 内检索时合并上游 canonical 知识和当前 Fork 增量，主知识库保持稳定。

检索阶段围绕 Wiki 页面、语义召回、关键词召回、Fork scope 和图谱扩展生成可追溯的 `context_pack`。上层应用和 Agent 可以获得稳定的知识对象、关系解释、隔离增量和引用证据，同时通过简单的 API 完成上传、查询、检索和运维集成。

## 标准流程

### 步骤 1：启动服务

默认 Docker Compose 模板会启动 API Server、Admin Web、Worker、PostgreSQL、Redis 和内置 OCR 服务。资源受限的服务器可以使用 optional-OCR 模板先不启动 OCR。

### 步骤 2：创建知识库

在主页创建知识库，填写名称、slug、用途、输出语言和基础 schema。知识库是最顶层的数据集，后续资料、Wiki 页面、图谱、版本、Webhook 和 Fork 都归属于它。

### 步骤 3：导入资料

可以通过后台上传文件，也可以通过 OpenAPI 提交文件、文本、URL 或 Fork-owned submission。资料进入系统后会保存原始对象、解析结果、媒体资源和内容 hash。

### 步骤 4：等待入库任务完成

入库任务会依次经过解析、分析、生成、合并、索引和完成。任务完成的含义是 Wiki 页面、系统页、图谱、索引、版本和检索就绪状态都已经更新。

### 步骤 5：检查知识资产

查看 Wiki 页面、Graph View、版本和 Knowledge Check。这里看到的是编译后的长期知识资产。

### 步骤 6：接入 Retrieve

在检索调试页面输入查询，检查候选页面、引用、图谱扩展、图片 caption 命中和 `context_pack`。确认效果后，把同样的请求形态复制到自己的服务端。

## 核心边界

| 边界     | FocoContext 负责                        | 接入方负责                         |
| -------- | --------------------------------------- | ---------------------------------- |
| 用户产品 | 返回可追溯上下文                        | 对话、页面、业务流程和最终回答     |
| 知识资产 | Wiki 页面、图谱、引用、版本             | 决定哪些资料进入哪个知识库         |
| 检索结果 | 候选、引用、`context_pack`、图谱说明    | 组织提示词、调用业务模型或 Agent   |
| 权限     | API Key、默认管理账号、资源存在性检查   | 终端用户身份、租户权限、业务授权   |
| 更新方式 | 上传、Source Watch、Fork 提交和版本记录 | 何时提交资料、如何确认外部生成内容 |

## 什么时候使用管理后台

管理后台适合做部署验证、数据观察、运维排障和开发调试。

| 页面            | 常用动作                                   |
| --------------- | ------------------------------------------ |
| 主页            | 创建知识库、查看总体状态                   |
| 资料            | 上传文件、查看资料状态、删除影响预览       |
| 任务            | 查看阶段事件、重试失败任务                 |
| Wiki 页面       | 阅读编译后的页面和来源引用                 |
| Graph View      | 查看关系、社群、桥接页面和图谱洞察         |
| Knowledge Check | 查看孤立页、断链、缺源、重复候选等质量信号 |
| 版本            | 查看 Change Set、页面版本、回滚记录        |
| 检索调试        | 验证 Retrieve 请求和上下文预算             |
| 设置            | 查看部署配置、模型、存储和运行状态         |

## 什么时候使用 OpenAPI

生产接入应优先通过 OpenAPI。典型调用链路如下：

```text
Create Knowledge Base -> Upload Source -> Poll Job -> Inspect Pages -> Retrieve -> Handle Webhook
```

如果你要把 FocoContext 接入自己的 SaaS、Agent 服务或内部平台，应在自己的后端保存知识库 ID、API Key、任务 ID 和必要的资源 ID，不建议让浏览器客户端直接持有 Bearer API Key。

## 生产注意事项

- 管理员账号、API Key、模型密钥、S3 密钥都应写入 `.env` 或部署平台密钥系统。
- 原始资料和派生资产存储在 S3-compatible 对象存储中，删除动作会进入异步清理。
- Source Watch 扫描外部位置时，应先手动扫描并查看 scan history，再开启定时扫描。
- 大文件、OCR、图片 caption 和模型生成都会消耗 Worker 资源，应根据服务器配置调整并发。
- UI 语言只影响后台展示；知识库输出语言由知识库配置决定。
