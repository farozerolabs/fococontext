# 资料入库

## 引言

本页解释资料页和上传弹窗。Source 是证据层；编译后的 Wiki 页面才是长期知识资产。

## 步骤 1：选择导入方式

| 方式            | 适合场景                                   | 入口              |
| --------------- | ------------------------------------------ | ----------------- |
| 文件上传        | 管理员手动导入 PDF、DOCX、Markdown、表格等 | 后台资料页        |
| 文本提交        | 开发者提交已生成或人工确认的内容           | OpenAPI           |
| URL 提交        | 服务端导入单个网页或远程文件               | OpenAPI           |
| Source Watch    | 周期扫描目录、S3、URL 列表或 Git           | 后台 Source Watch |
| Fork submission | 终端用户或工作区隔离提交新增知识           | OpenAPI           |

## 步骤 2：上传文件

在资料页点击上传资料。上传弹窗会显示文件列表、进度和每个文件的状态。

可选字段：

| 字段                | 说明                                                                                                                |
| ------------------- | ------------------------------------------------------------------------------------------------------------------- |
| 归档路径 / 虚拟目录 | 用于逻辑归类，例如 `policies/2026/`                                                                                 |
| 元数据标签          | 用于筛选和追溯，例如 `department=operations`、`source=manual`                                                       |
| 知识库              | 文件要进入的目标 Knowledge Base                                                                                     |
| 上传模式            | 自动模式沿用运行时阈值策略；内置上传强制使用后台 multipart API；直链上传在对象存储运行配置就绪后使用 upload session |

上传成功后，后台会创建 Source Document，并把原始文件写入 S3-compatible 存储。object key 使用资源 ID 和随机后缀，避免同名文件互相覆盖。

上传模式只影响原始文件的传输方式。入库排队、解析、Wiki 生成、图谱更新、索引和版本记录仍会完整执行。

直链上传就绪状态由运行时状态中的 `limits.upload.directUpload.ready` 表示。大文件走 upload session 前，应从 `/health` 或系统设置读取这个 canonical 字段。

## 步骤 3：理解支持格式

| 类型                    | 处理方式                                           |
| ----------------------- | -------------------------------------------------- |
| Markdown / text         | 直接抽取结构和正文                                 |
| PDF                     | 先解析文本、页面和图片；低文本或扫描内容可启用 OCR |
| DOCX / PPTX / XLSX      | 转换为结构化 Markdown、表格和媒体资产              |
| CSV / JSON / YAML / XML | 保留结构并转换为可分析文本                         |
| HTML                    | 抽取正文、链接和标题结构                           |
| 图片                    | 作为媒体资产保存，可选生成 caption                 |

解析器会输出 normalized Markdown、plain text、locator、表格、警告和媒体资产。解析警告不一定代表失败，应结合任务状态判断。

## 步骤 4：查看资料列表

资料列表按更新时间倒序显示，和任务页一致。建议重点看：

| 列       | 含义                                               |
| -------- | -------------------------------------------------- |
| 名称     | 原始文件名或提交标题                               |
| ID       | Source Document ID，用于 API 排查                  |
| 状态     | queued、processing、completed、failed、deleting 等 |
| 当前阶段 | 最近任务阶段                                       |
| 更新时间 | 最近一次上传、处理或清理时间                       |
| 最新任务 | 关联的 Ingest Job                                  |

点击一条资料可以在右侧查看详情，包括原始元数据、解析结果、媒体资产、caption 状态、任务时间线和删除影响。

## 步骤 5：等待编译完成

上传完成只代表原始文件进入对象存储。只有关联任务 completed 后，才代表 Wiki 页面、图谱、索引和版本已经更新。

如果任务失败：

1. 打开任务详情。
2. 查看失败阶段和错误消息。
3. 检查模型、解析器、OCR、S3 或索引配置。
4. 修复后点击重试。

## 步骤 6：删除资料

删除资料前可以先预览影响。预览会告诉你可能受影响的页面、索引、图谱边、版本和对象存储资源。

删除后后台会立即隐藏普通列表中的资料，并异步执行级联清理。正在运行的相关任务会被取消或阻止继续写入，避免删除后又被旧 Worker 写回。

## 生产注意事项

- 大文件在 `limits.upload.directUpload.ready` 为 `true` 时建议使用直链上传；多文件大小差异较大时建议使用自动模式。
- 同名文件会保存在不同 S3 对象中，后台仍显示原始文件名，便于管理员理解。
- 图片 caption 失败应和文本入库隔离；可以单独重试 caption。
- 对扫描 PDF，先确认 OCR 服务和并发配置，再批量上传。
- Source 是证据输入；最终检索主体是 Wiki 页面。
