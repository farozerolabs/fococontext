# Fork 提交

## 引言

Fork 用于隔离终端用户、团队空间或业务工作区产生的新增知识。它让上游 Knowledge Base 保持干净，同时允许下游空间持续补充自己的资料。

## 为什么需要 Fork

服务端产品通常会被多个开发者和终端用户共同使用。Fork 为每个客户、工作区或用户提供基于同一个上游知识库的隔离覆盖层。主知识库保持稳定，每个 owner 可以持续补充自己确认过的私有资料。

| 场景                   | 推荐方式                  |
| ---------------------- | ------------------------- |
| 公共资料库             | 写入主 Knowledge Base     |
| 某个客户自己的补充资料 | 写入客户 Fork             |
| 某个终端用户的个人资料 | 写入用户 Fork             |
| 临时实验资料           | 写入实验 Fork，测试后删除 |

主知识库仍然负责公共、稳定、经过管理员确认的资料。Fork 负责某个 owner 的私有增量。这样可以让同一个上游资料库服务多个客户、团队或终端用户，并保持不同 owner 的新增内容互相隔离。

## 步骤 1：解析 Fork

外部应用先调用 resolve fork 接口，用上游 Knowledge Base ID、owner 类型和 owner key 获取稳定 Fork。

owner key 应由接入方生成并保存，例如用户 ID、团队 ID、项目 ID。FocoContext 不负责管理你的终端用户体系。

## 步骤 2：提交资料

Fork-owned submission 可以提交人工确认文本、业务系统生成内容或用户上传资料。提交后仍走同一套解析、分析、生成、合并、索引和版本流程。

| 字段        | 说明                             |
| ----------- | -------------------------------- |
| fork_id     | 目标 Fork                        |
| title       | 提交标题                         |
| content     | 要写入的正文或摘要               |
| source type | 外部系统、用户上传、自动化流程等 |
| metadata    | 业务侧 ID、标签、来源 URL        |

## 步骤 3：等待任务完成

提交成功后会返回任务信息。接入方应轮询任务或监听 Webhook，等任务 completed 后再对终端用户开放新的检索结果。

建议保存这些字段：

| 字段                 | 用途                       |
| -------------------- | -------------------------- |
| submission_id        | 追踪本次外部提交           |
| job_id               | 查询入库进度               |
| source_id            | 查看证据和解析结果         |
| change_set_id        | 查看本次提交影响了哪些页面 |
| knowledge_version_id | 标记 Fork 范围内可检索版本 |

## 步骤 4：带 Fork 范围检索

检索时传入 fork scope。系统会在上游知识和 Fork 自有知识之间合并可见上下文，Fork 内容保留在自己的隔离范围内。

| 检索范围        | 结果                             |
| --------------- | -------------------------------- |
| upstream only   | 只查主知识库                     |
| fork scope      | 查主知识库 + 当前 Fork 增量      |
| fork-owned only | 只查当前 Fork 自有内容，适合排查 |

外部应用应在自己的权限系统里决定用户能访问哪个 Fork。FocoContext 根据请求中的 fork scope 计算知识可见范围，业务侧登录态和授权判断仍由外部应用负责。

## 步骤 5：删除 Fork 或提交

删除 Fork 应异步清理它的资料、页面、索引和对象存储资源。上游 Knowledge Base 不应受影响。

删除前确认：

- 是否有业务用户仍在使用该 Fork。
- 是否需要导出 Fork 自有 Markdown。
- 是否存在正在运行的提交任务。

## 生产注意事项

- owner key 不应包含明文敏感信息，建议使用内部稳定 ID。
- Fork scope 是知识可见范围。外部应用仍要自己校验终端用户是否能访问对应 Fork。
- Fork 提交不包含托管外部研究能力；外部应用自己决定生成、确认和提交内容。
- 如果外部应用同时维护缓存，应在收到任务完成事件后再刷新该 Fork 的检索缓存。
- 如果 Fork 被删除，继续使用旧 fork ID 的请求应被视为资源已不存在，业务侧需要重新 resolve。
- 对账时可以按 owner key、fork ID、submission ID 和 job ID 建立链路，方便定位某个用户提交的内容如何影响检索结果。
